## Current Implementation State

### Completed Features

1. **Bidirectional Sync for All Providers**
   - Dropbox/WebDAV: Uses KOReader's `SyncService.sync()` with ETag-based conflict detection
   - FTP: Custom `FtpSync` module with timestamp-based (MDTM) conflict detection
   - All providers now support: upload, download, conflict detection, three-way merge

2. **Core Modules**
   - `main.lua`: Plugin UI, menu integration, settings management
   - `syncengine.lua`: Orchestrates sync, routes to appropriate sync method
   - `cloudprovider.lua`: Unified API for cloud storage operations
   - `ftphelper.lua`: FTP protocol operations (MDTM, LIST, MKD, upload, download)
   - `ftpsync.lua`: FTP bidirectional sync with cache-based comparison
   - `synccache.lua`: JSON persistence for sync state and conflict resolution

3. **Sync Logic**
   - Three-way merge for Lua table files (bookmarks, annotations, settings)
   - Local changes take precedence on direct conflicts
   - Deleted files propagate correctly (delete local → delete remote, and vice versa)
   - Clock skew handled by comparing deltas, not absolute timestamps

## Architecture Decisions

### Why FtpSync is Separate from SyncService

KOReader's `SyncService` only supports Dropbox and WebDAV because it relies on:
- **ETags**: HTTP-specific content hashes for collision detection
- **If-Match headers**: HTTP-specific conditional uploads
- **412 status codes**: HTTP-specific conflict responses

FTP lacks these HTTP semantics, so we implemented `FtpSync` which uses:
- **MDTM command**: Get file modification time (YYYYMMDDhhmmss format)
- **Local sync cache**: Store mtime at last sync for three-way comparison
- **Timestamp deltas**: Compare `(now - cached)` to detect changes, avoiding clock skew

### File Responsibilities

```
main.lua
  └─> syncengine.lua
        ├─> SyncService (Dropbox/WebDAV)
        │     └─> KOReader's apps/cloudstorage/syncservice.lua
        └─> FtpSync (FTP)
              ├─> ftphelper.lua (protocol operations)
              └─> synccache.lua (state persistence)
```

### Sync Cache Structure

Located at `~/.config/koreader/settings/supersync_cache.json`:

```json
{
  "version": 1,
  "servers": {
    "my_ftp_server": {
      "last_sync_time": 1706140800,
      "files": {
        "/KOReader-SuperSync/book.sdr/metadata.pdf.lua": {
          "local_mtime": 1706140000,
          "remote_mtime": 1706139500,
          "synced_at": 1706140800
        }
      },
      "cached_content": {
        "/KOReader-SuperSync/book.sdr/metadata.pdf.lua": "{...json...}"
      }
    }
  }
}
```

## Known Limitations

1. **MDTM Not Universally Supported**
   - Some old FTP servers don't support MDTM command
   - FtpSync falls back to upload-only mode with warning
   - User sees: "FTP server does not support MDTM - falling back to upload-only"

2. **Binary File Conflicts**
   - Three-way merge only works for Lua table files
   - Binary files (screenshots, etc.) use last-write-wins
   - This is acceptable since .sdr files are mostly Lua

3. **No LIST Timestamp Parsing Fallback**
   - If MDTM fails, we don't parse LIST output for timestamps
   - LIST format varies wildly between servers
   - Could be added but complexity vs value tradeoff

4. **Clock Skew Window**
   - Using timestamp deltas reduces but doesn't eliminate clock issues
   - If device clocks differ by hours, first sync may misdetect changes
   - Subsequent syncs work correctly

## Testing Notes

### To Test FTP Bidirectional Sync

1. Set up an FTP server that supports MDTM (vsftpd, FileZilla Server, ProFTPD)
2. Configure server in KOReader: Tools → Cloud Storage → Add FTP
3. Configure SuperSync: Super Sync → Settings → Select FTP server
4. Create some bookmarks/annotations on device
5. Sync: Super Sync → Sync now
6. Modify files on server (or another device)
7. Sync again and verify changes downloaded

### To Test Conflict Resolution

1. Sync from Device A
2. Add bookmark on Device A (don't sync yet)
3. Add different bookmark on Device B, sync
4. Sync Device A
5. Verify both bookmarks present (merged)

### To Test MDTM Fallback

1. Use an FTP server without MDTM support
2. Attempt sync
3. Verify warning message appears
4. Verify upload-only mode works
